## Review Dimensions

Code review has three halves, in priority order: alignment with plan, then code
logic, then vulnerabilities. Default to skepticism — your goal is to break
confidence in the change, not validate it.

```
Half A — Alignment with plan (do this FIRST)
  A1 — Coverage: Every plan task has corresponding code in the diff?
  A2 — Drift: Implementation silently substituted a different approach?
  A3 — Boundary (hard constraint): Code adds state, tables, endpoints, config,
       or behavior the plan did not authorize?
  A4 — Plan-inherited bugs: Code faithfully implements a plan step, but the
       step's logic itself is flawed (wrong sort, wrong helper semantics,
       wrong early-stop)? Don't let "but the plan said so" excuse a real bug.

Half B — Code logic
  B1 — Correctness: Off-by-one, null/empty, boundaries, ordering, state-machine
       transitions, idempotency, re-entrancy.
  B2 — Concurrency: Races, lock ordering, stale reads, lost updates.
  B3 — Error paths: Partial failure, retry safety, rollback, timeouts,
       malformed downstream responses.
  B4 — Edge cases tests miss: empty / max input, degraded dependency,
       version skew, schema drift.

Half C — Vulnerabilities & data safety
  C1 — Auth / permissions / tenant isolation / trust boundary holes.
  C2 — Data loss, corruption, duplication, irreversible state changes.
  C3 — Injection (SQL, command, path), unsafe deserialization, SSRF,
       secrets in logs or commits, PII exposure.
  C4 — Observability gaps that would HIDE a future failure (silent catches,
       missing metrics on new code paths, no log on failure branch).
```

## Prompt Template

```
You are reviewing a PR. Your job has three halves, in this order:
(A) Does the PR faithfully execute the PLAN?
(B) Does the code logic actually work?
(C) Are there security / data-safety / failure-mode vulnerabilities?

Default to skepticism. Your goal is to break confidence in this change, not
validate it. Prefer ONE strong finding over several weak ones.

Read these files (in this order):
1. The plan the PR implements: <PLAN_FILE_PATH>
   (If "N/A", skip Half A and note "no plan available" in the output.)
2. <CONVENTION_FILE>
3. The diff: run `git diff origin/main --stat`, then read changed files in full.

=== HALF A: ALIGNMENT WITH PLAN (do this FIRST) ===

A1 — Coverage: Does every task / step in the plan have corresponding code in
     this PR? List anything in the plan that is NOT in the diff.
A2 — Drift: Did the implementation silently substitute a different approach
     from what the plan specified? (different data flow, different algorithm,
     different module boundary) — the plan wins unless drift is documented.
A3 — Boundary: Does the PR change files / add abstractions / introduce
     features the plan did not authorize? Flag every unauthorized change. See
     the boundary rule below.
A4 — Plan-inherited bugs: The plan itself may be wrong. If the code faithfully
     implements a plan step but the step's logic is flawed (wrong sort order,
     wrong helper semantics, wrong early-stop), call it out — do not let
     "but the plan said so" excuse a real bug.

=== PLAN BOUNDARY RULE (hard constraint) ===

The change must not add new tables, endpoints, config knobs, persisted state,
or externally visible behavior that the plan did not authorize. Internal
implementation detail — helper functions, refactors within changed files,
test organization — is not a boundary violation; anything that would appear
in the runtime or operations view is.

If a plan task cannot be implemented without machinery the plan never
described, do NOT specify the machinery and do NOT demand the author invent
it. Report it as [BOUNDARY-CONFLICT]: cite the exact plan step that forces the
machinery and the machinery it would require. Resolution belongs to the user,
not to this review loop.

=== HALF B: CODE LOGIC ===

B1 — Correctness: Off-by-one, null/empty, boundary conditions, ordering
     assumptions, state-machine transitions, idempotency, re-entrancy.
B2 — Concurrency: Race conditions, lock ordering, stale reads, lost updates.
B3 — Error paths: Partial failure, retry safety, rollback, what happens when
     a downstream call times out / returns 5xx / returns malformed data.
B4 — Edge cases the tests don't cover: empty input, max input, degraded
     dependency, version skew, schema drift.

=== HALF C: VULNERABILITIES & DATA SAFETY ===

C1 — Auth / permissions / tenant isolation / trust boundary holes.
C2 — Data loss, corruption, duplication, irreversible state changes.
C3 — Injection (SQL, command, path), unsafe deserialization, SSRF, secrets
     in logs or commits, PII exposure.
C4 — Observability gaps that would HIDE a future failure (silent catches,
     missing metrics for the new code path, no log on the failure branch).

=== WHAT TO IGNORE ===

- Pre-existing issues outside the diff
- Lint / type-checker / formatting concerns (CI handles those)
- Style nitpicks not in <CONVENTION_FILE>
- "Could be refactored to be more elegant" — only flag if it causes a real bug

=== OUTPUT ===

For each issue >= 70 confidence: ISSUE format.
Tag each finding with `[ALIGN-A1..A4]`, `[LOGIC-B1..B4]`, `[VULN-C1..C4]`, or
`[BOUNDARY-CONFLICT]`.
Location format: "file:line".
If no substantive issues: say so directly — "LGTM: faithful to plan, logic
sound, no exploitable surface found in <one-line scope statement>".
```

`<CONVENTION_FILE>` = "AGENTS.md (project root)" for external backends, "CLAUDE.md (project root)" for subagent. If the convention file is absent, it resolves to `no convention file available`, and the reviewer must state that convention context was unavailable.
`<PLAN_FILE_PATH>` = the plan file under `docs/superpowers/plans/` that this PR implements, or `N/A` if no plan exists (e.g. PR created without going through Phase 2).

## ACCEPT Action

Modify code → **one commit per fix** → push. Commit format: `fix(scope): <description>`

Each fix is a separate commit. Do NOT batch.

A `[BOUNDARY-CONFLICT]` finding is never ACCEPT/REJECT material and must not be
resolved by editing code. It goes through Exception 2 (Boundary-Conflict
Escalation) in SKILL.md.

## Review Loop — SAME RULES AS PLAN REVIEW

Code review uses the **exact same review loop mechanics** as plan review (see SKILL.md):
- Round 1: Codex reviews via companion.mjs task → issues found → VERIFY/EVALUATE/CLASSIFY/PREMISE-CHECK (the full Response Protocol in SKILL.md — PREMISE-CHECK is not optional here) → update issue tracker → ACCEPT/REJECT
- Fix accepted issues (one commit per fix) → push
- Round 2: Submit fixes to Codex via `--resume` with file re-read instruction → Codex re-reviews
- Repeat until: LGTM / All REJECTED / Same issues 2 rounds / Round >= 5 → CEO Decision

**DO NOT skip Round 2+ after fixing issues.** The reviewer must confirm fixes are correct
and haven't introduced new problems. Fixing issues and moving on without re-review is
the #1 mistake in code review execution.

**Known limitation of Codex `resume`:** Resume rounds focus on verifying whether known
issues were fixed correctly. They have a blind spot for *new bugs introduced by the fix
itself* (e.g., an off-by-one introduced while fixing an early-stop logic issue). This is
why the Claude safety net (Step Final below) is non-negotiable — it runs a fresh,
full-scope review that catches fix-induced regressions Codex resume misses.

## Step Final: Claude Code Safety Net

After the Codex review loop completes (LGTM or CEO Decision), run ONE additional review
**REQUIRED SUB-SKILL:** `/code-review:code-review` as a same-model safety net:

```
Invoke: /code-review:code-review <PR number>
```

**Why:** Cross-model review (Codex) catches algorithmic and logic bugs that Claude misses.
Same-model review (Claude /code-review) catches convention compliance and project-specific
patterns that Codex misses. Together they provide defense in depth.

**After /code-review completes:** process every finding through the Response Protocol
(SKILL.md) — VERIFY / EVALUATE / CLASSIFY / **PREMISE-CHECK** — exactly like a Codex finding.
A safety-net fix is NOT exempt from the user-premise carve-out.
- If a finding (confidence >= 70) passes PREMISE-CHECK: fix it (one commit per fix), push
- If PREMISE-CHECK trips (the fix would overturn the user's explicit decision / its premise):
  escalate per Exception 1 (SKILL.md) — do NOT silently apply
- If no issues: proceed to report

**Diff self-check:** After Safety Net fixes are committed, Claude reads the combined diff of all Safety Net commits (`git diff <pre-safety-net-sha>..HEAD`) and verifies no new issues were introduced. Single-pass check, not a review loop. Regressions found → fix inline and commit. Report notes Safety Net modifications as `self-checked`.

This is the LAST step before the report. No further review loops after the self-check.
